# Documentation Standards

## Code Comments

- **Explain why, not what**: Code shows what, comments explain rationale
- **Document assumptions**: Note any non-obvious assumptions
- **Mark TODOs clearly**: Use `TODO(username):` with explanation
- **Keep comments up to date**: Update comments when code changes
- **Avoid redundant comments**: Don't comment obvious code

## Function/Module Documentation

- **JavaScript/TypeScript**: Use JSDoc with `@param`, `@returns`, `@throws`
- **Python**: Use docstrings (Google or NumPy style) with types
- **Rust**: Use doc comments (`///`) with examples in doc tests
- **Include examples**: Show common usage patterns
- **Document errors**: Specify what errors/exceptions can occur

## README and Project Docs

- **Clear project description**: What it does and why it exists
- **Quick start guide**: Get users running quickly
- **Installation instructions**: All dependencies and setup steps
- **Usage examples**: Common use cases with code samples
- **Contributing guidelines**: How to contribute (if applicable)
- **API documentation**: Auto-generated where possible
